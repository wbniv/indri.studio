# One-time setup

Manual operations that need doing once per repo / account before the automated tag-push deploy ([`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml)) will work. On a fresh clone, work through every section below before pushing your first `v*` tag.

The project follows the **per-project new-account discipline** — every service gets fresh credentials for indri.studio, not reused from rapid-raccoon or other sibling projects. See the plan's "Credentials & accounts" section.

## 1. Cloudflare account + zone

Create or designate a dedicated **Cloudflare Account** for indri.studio (separate from any rapid-raccoon Account on your login). Then add the `indri.studio` zone to that Account and ensure it shows as **Active**.

To check status: <https://dash.cloudflare.com/> → **Account home** → **Domains** tile. A **green checkmark** next to `indri.studio` = Active.

If you see something other than a green check:

- **Pending** — Cloudflare is waiting for the domain's nameservers to point at theirs. Click into the zone; the Overview page shows which nameservers to set at your registrar.
- **Initializing / Moved / Read Only / Deactivated** — click into the zone; the Overview or Notifications tab explains.

If `indri.studio` is registered *through* Cloudflare Registrar, the zone is set up with Cloudflare nameservers automatically and goes Active without a registrar step. Recommended.

## 2. Cloudflare API token (project-scoped)

Create a token narrowed to indri.studio only. **Do NOT reuse the rapid-raccoon token** or your global API key.

- Open: <https://dash.cloudflare.com/profile/api-tokens>
- Click **Create Token** → **Edit Cloudflare Workers** template → **Use template**.
- In **Account Resources**: pick the dedicated indri.studio Account from step 1.
- In **Zone Resources**: include `indri.studio` only.
- Click **Continue to summary** → **Create Token**.
- Copy the token. **It is shown only once** — if you lose it, revoke and recreate.

Long-term, this token is itself managed by Terraform in [`infrastructure/cloudflare/iam-self/`](../infrastructure/cloudflare/iam-self/), self-narrowed via a bootstrap token.

## 3. Cloudflare account ID

Fastest path: look at the URL after login on <https://dash.cloudflare.com/>. It redirects to `dash.cloudflare.com/<ACCOUNT_ID>/home/overview` — the 32-character hex string is your account ID. Or click into any zone; the right-hand sidebar shows the Account ID with a copy button.

## 4. Push secrets to SSM (source of truth)

SSM is the source of truth for all project secrets. Local `.env` files are read-only caches regenerated by `task secrets-pull`.

```sh
aws ssm put-parameter \
  --name /indri-studio/cloudflare/api_token \
  --type SecureString \
  --value "<token from step 2>"

aws ssm put-parameter \
  --name /indri-studio/cloudflare/account_id \
  --type String \
  --value "<account ID from step 3>"
```

Or via `task secrets-bootstrap` once that script is in place.

## 5. GitHub repo + secrets

Create the GitHub repo under the dedicated **indri-studio** GitHub organization (see "Credentials & accounts" in the plan). Don't push to your personal namespace or to rapid-raccoon-site's org.

The deploy workflow reads two repo secrets:

```sh
# Pull from SSM, then push into GitHub
aws ssm get-parameter --name /indri-studio/cloudflare/api_token \
  --with-decryption --query Parameter.Value --output text \
  | gh secret set CLOUDFLARE_API_TOKEN

aws ssm get-parameter --name /indri-studio/cloudflare/account_id \
  --query Parameter.Value --output text \
  | gh secret set CLOUDFLARE_ACCOUNT_ID
```

Or via the UI: `https://github.com/indri-studio/indri.studio/settings/secrets/actions` → **New repository secret** for each. (Replace `indri-studio/indri.studio` with the actual GitHub org/repo path.)

## 6. Terraform bootstrap

```sh
cd infrastructure/cloudflare/bootstrap
terraform init && terraform apply
```

Sets up the TF state backend (shared S3 bucket, same as parking-space / bumper2bumper / finding-your-way).

Then:

```sh
cd ../iam-self    && terraform init && terraform apply    # narrow the token
cd ../global      && terraform init && terraform apply    # zone, Workers domain bindings, redirects
```

After `global` applies, `indri.studio` and `www.indri.studio` are bound to the Worker and the canonical-host redirects are live.

## 7. First-deploy sanity check

The very first custom-domain deploy can hit a "zone not active" error while Cloudflare's internal check catches up. Failed deploys change nothing; the previous workers.dev deploy keeps serving. Wait a few minutes and re-run from the Actions tab rather than cutting a new tag.

## Local deploy (no CI)

Fallback for throwaway test deploys before tagging.

```sh
pnpm wrangler login                          # one-time per machine, OAuth flow
task build && pnpm wrangler deploy            # or: task deploy
```

This uses your own Cloudflare credentials (not the CI token).
