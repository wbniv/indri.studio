# Emulator HUD overscan-crop fix (`/blossom` value bar clipped)

**Date:** 2026-06-27
**Status:** Fixed & live on both sites. biohack briefly regressed (a `space-invaders` bundle re-sync
reverted its vendored copy to `yoff=8`) and was **re-fixed** (`fc918d5`, `v1.0.76`); see
[§Regression](#regression--biohack-re-synced-back-to-yoff8). The durable fix lives in the
`bsnes-jg-wasm` source, which `deploy-bundle.sh` copies into the bundle, so future syncs carry it.
**Type:** Bug fix.

Related: [llvm-mos emulator embed](2026-06-25-llvm-mos-emulator-embed.md) §"Aspect fixed" (the
crop logic this corrects).

---

## Symptom

On [https://indri.studio/blossom/](https://indri.studio/blossom/) (and the sibling
[https://biohack.net/blossom/](https://biohack.net/blossom/)), the in-browser SNES player's **top
value bar** (`CLASSIC A7.17 B8.44 C2.56`) rendered jammed flush against the top edge of the canvas
with the **glyph tops shaved off**, and a fat black band sat below the bottom control bar. The
page's own static screenshots (`/blossom/default.png`) showed the bar correctly, with an 8‑pixel
black margin above it — so the live player was inconsistent with the captures beside it.

The Mandelbrot demo on `/apps/llvm-mos-65816/` shared the same player but never showed the problem
(its content is a centred fractal with no top‑edge HUD), which is why it went unnoticed.

## Root cause

The player's frame blitter, `present()` in the shared `app.js`, cropped the framebuffer with:

```js
var yoff = h >= 232 ? 8 : 0;   // skip the top NTSC overscan (active picture starts at line 8)
```

This assumes an 8‑line top NTSC overscan band. It is **wrong for this bsnes‑jg build**: the active
224‑line picture sits **flush at the top** of the 512×240 buffer. Skipping 8 rows takes raw rows
`[8, 232)` instead of `[0, 224)`, shoving every frame up by 8 px — invisible on the centred
Mandelbrot, but it ate the blossom ROM's top value bar and left a black band at the bottom.

### Evidence (headless probe of the actual WASM core, frame 700)

Running the real `bsnes_jg.wasm` in Node (loaded as `.cjs` to bypass the repo's `type: module`
which makes `require()` parse the Emscripten glue as ESM and return `{}`):

```
RAW: w=512 h=240 pitch=512 step=2 ow=256
blossom:        first lit raw row=8   last lit raw row=221   (rows 0–7 are black)
mandel-display: first lit raw row=8   last lit raw row=232
```

The gate's reference PNGs — which `present()` must reproduce — all capture raw rows **`[0, 224)`**
(8 black rows on top, then the picture; the bottom is what gets cropped):

```
indri  preview/blossom.png : 256x224 first-lit-row=8
biohack preview/blossom.png: 256x224 first-lit-row=8   (byte-identical to indri's)
indri  /blossom/default.png: 256x224 first-lit-row=8
mandel-jg.png (gate ref)   : 256x224 first-lit-row=8
```

So both reference PNGs use `yoff=0`, while the live `present()` used `yoff=8` — mis‑framing **both**
ROMs by 8 px; only blossom's top bar made it visible.

## Fix

Set `yoff = 0` in `present()`. Cosmetic only — the crop never touched the WRAM fidelity assert
(`0x9047` / `0x9103`), so the gate self‑check is unaffected.

```js
var yoff = 0;   // active picture sits flush at the buffer top; jgxcheck refs capture raw rows [0,224)
```

## Regression — biohack re-synced back to `yoff=8`

After this fix shipped, biohack's per-site copy was **clobbered back to `yoff=8`** by a later
unrelated commit:

```
$ git -C ~/SRC/biohack.net log --oneline -4
957c2cf space-invaders: complete game build (shields + score/lives HUD), CRC 0x9D57
c20b62e space-invaders: playable in-browser SNES Space Invaders page   ← re-synced public/play/ bundle
3f9c66e play/app.js: don't skip 8-line "overscan" — it clipped /blossom's HUD   ← the fix (v1.0.74)
04a42bb blossom: playable in-browser SNES demo at /blossom

$ for c in 3f9c66e c20b62e HEAD; do git -C ~/SRC/biohack.net show $c:public/play/app.js | grep -m1 yoff; done
    var yoff = 0;                 # 3f9c66e (fix)
    var yoff = h >= 232 ? 8 : 0;  # c20b62e (clobbered)
    var yoff = h >= 232 ? 8 : 0;  # HEAD (live = clipped again)
```

`c20b62e` (adding the Space Invaders page) re-copied the shared player bundle into
`public/play/app.js`, overwriting `3f9c66e`. biohack.net/blossom is **clipped again** on live.

**Root lesson — the per-site copy is the wrong place to fix this.** `app.js` is vendored into each
site (`indri.studio/public/apps/llvm-mos-65816/play/`, `biohack.net/public/play/`) from the
`bsnes-jg-wasm` bundle. A hand-patch to a vendored copy is undone by the next bundle sync. The
durable fix lives in the **sync source** (`bsnes-jg-wasm/web/app.js` — done, `aaacbae`); each site
must then be **re-synced from the fixed bundle**, not hand-edited. Note `bsnes-jg-wasm/dist-bundle/`
is gitignored, so confirm `deploy-bundle.sh` regenerates it from `web/app.js` before any re-sync.

**Resolved (`fc918d5`, `v1.0.76`).** Re-applied `yoff=0` to biohack's `public/play/app.js` and added
a "vendored copy — keep in sync with `bsnes-jg-wasm/web/app.js`" note in-line so the next bundle pull
is less likely to silently revert it. Verified live at 125% zoom (21 px top margin, glyph tops
intact). The bundle source is now correct end-to-end: `bsnes-jg-wasm/web/app.js` is `yoff=0`
(`aaacbae`) and `deploy-bundle.sh:46` does `cp web/app.js dist-bundle/app.js`, so a fresh bundle —
and therefore any future re-sync of either site — carries the fix. (The one residual sharp edge:
`dist-bundle/` is gitignored, so a stale, un-rebuilt bundle could still clobber a site; rebuild the
bundle before syncing.)

## Notes / follow‑ups

- **Stale tab caveat.** The indri tab in the bug report showed the *pre‑centering* 890 px layout, so
  it was a cached older build. HTML is `no-store`, but bfcache can survive Ctrl+Shift+R — a full
  close+reopen is the reliable refresh. Cloudflare returned `cf-cache-status: HIT` on the `no-store`
  HTML; if a stale edge copy ever recurs, purge the `/blossom/` URL via the API.
- **Verification‑tooling lesson.** A first measurement pass falsely reported biohack "still clipped"
  because the HUD‑locator latched onto the page's lede text instead of the player. Locate the player
  by its black‑screen run *first*, then find the HUD within it — don't trust a bare colour match
  against the whole frame.
- **Pre‑existing biohack CI gate.** `v1.0.72`–`v1.0.74` all show the Actions run as "failed" purely
  on the post‑deploy Lighthouse **Threshold check**; the deploy itself is green. Worth a separate
  look at whether the threshold needs adjusting, but it does not block deploys.
